## 1. 项目概述

### 1.1 项目背景
随着智能手机普及，通讯录数据分散在不同设备，存在数据丢失、重复联系人、多设备不同步等问题。CloudSyncContacts 旨在提供安全、可靠、智能的通讯录云端同步方案，支持 iOS 和 Android 双向同步，并具备智能去重合并能力。

### 1.2 项目目标
- 实现 iOS 和 Android 手机通讯录安全上传至云端
- 支持云端与本地通讯录的双向比对同步
- 自动检测并合并重复联系人，支持手动确认
- 提供完整的冲突解决机制
- 保证数据安全性和隐私保护
- 版本迭代开发，每个版本可独立验证

### 1.3 适用范围
| 维度 | 说明 |
|---|---|
| 支持平台 | iOS 15+ / Android 10+（API 29+） |
| 云端架构 | RESTful API + PostgreSQL + Redis + 对象存储 |
| 目标用户 | 普通消费者、企业用户 |
| 最大联系人数 | 免费版 5,000 / 付费版 50,000 |

## 2. 系统架构

### 2.1 整体架构
系统采用三层架构：客户端层、服务层、数据层。客户端负责本地通讯录读取与 UI 交互，服务层处理同步逻辑、冲突解决、去重合并，数据层提供持久化存储。

| 层级 | 技术组件 | 职责 |
|---|---|---|
| 客户端层 | iOS（Swift/SwiftUI）、Android（Kotlin/Jetpack Compose） | 本地通讯录访问、UI 交互、本地缓存 |
| 服务层 | Go/Gin + gRPC、Redis 缓存 | API 网关、同步引擎、去重算法、冲突解决 |
| 数据层 | PostgreSQL、MinIO、Elasticsearch | 联系人存储、全文检索、备份快照 |
| 基础设施 | Docker、K8s、Prometheus、Grafana | 容器编排、监控告警、日志 |

### 2.2 数据模型（联系人核心实体）
| 字段名 | 类型 | 说明 |
|---|---|---|
| contact_id | UUID | 全局唯一标识符 |
| user_id | UUID (FK) | 所属用户 |
| display_name | VARCHAR(255) | 显示名称 |
| given_name / family_name | VARCHAR(128) | 名 / 姓 |
| phone_numbers | JSONB | 电话数组 `[{type,value,label}]` |
| email_addresses | JSONB | 邮箱数组 |
| postal_addresses | JSONB | 地址数组 |
| organization / job_title | VARCHAR(255) | 公司 / 职位 |
| notes | TEXT | 备注 |
| photo_uri | VARCHAR(512) | 头像存储路径 |
| source_device_id | VARCHAR(128) | 来源设备标识 |
| local_id | VARCHAR(128) | 本地系统 ID 映射 |
| version | BIGINT | 乐观锁版本号 |
| sync_status | ENUM | `synced/pending/conflict/deleted` |
| hash | VARCHAR(64) | 内容指纹（SHA-256） |
| created_at / updated_at | TIMESTAMP | 创建 / 更新时间 |
| deleted_at | TIMESTAMP | 软删除时间戳 |

## 4. v0.1 基础上传

### 4.1 功能范围

#### 4.1.1 iOS 通讯录访问
- 使用 Contacts Framework（`CNContactStore`）读取本地通讯录
- 请求用户授权（`CNAuthorizationStatus`），处理拒绝/受限场景
- 批量读取联系人，分页加载（每次 200 条）
- 提取字段：姓名、电话、邮箱、地址、公司、职位、头像、备注

#### 4.1.2 Android 通讯录访问
- 使用 `ContactsContract` ContentProvider 读取本地通讯录
- 运行时权限申请（`READ_CONTACTS` / `WRITE_CONTACTS`）
- 使用 `CursorLoader` 异步加载，避免主线程阻塞
- 映射到统一数据模型

#### 4.1.3 云端 API
| 接口 | 方法 | 路径 | 说明 |
|---|---|---|---|
| 用户注册 | POST | `/api/v1/auth/register` | 邮箱/手机号注册 |
| 用户登录 | POST | `/api/v1/auth/login` | 获取 JWT Token |
| 批量上传 | POST | `/api/v1/contacts/batch` | 批量上传（最大 500 条/次） |
| 查询列表 | GET | `/api/v1/contacts` | 分页查询云端联系人 |
| 上传头像 | POST | `/api/v1/contacts/{id}/photo` | 上传头像至对象存储 |

## 5. v0.2 双向同步

### 5.1 同步策略
采用“时间戳 + 内容哈希”的增量同步：客户端发送 `last_sync_time` 与本地变更集，服务端返回服务端变更集，客户端合并后应用。

### 5.1.1 同步流程
1. 客户端计算本地变更（新增/修改/删除）
2. 客户端调用 `POST /api/v1/sync`，携带 `{last_sync_time, local_changes[]}`
3. 服务端查询 `last_sync_time` 之后的服务端变更
4. 服务端比对双方变更并检测冲突
5. 无冲突：自动合并并返回结果
6. 有冲突：返回冲突列表（v0.3 处理）
7. 客户端应用服务端变更到本地通讯录
8. 更新本地 `last_sync_time`

### 5.1.2 变更检测机制
| 平台 | 检测方式 | 说明 |
|---|---|---|
| iOS | `CNChangeHistoryFetchRequest` | iOS13+ 获取精确变更事件 |
| iOS（降级） | 全量 hash 比对 | 旧版通过 hash 差异检测变更 |
| Android | `ContentObserver` 监听 `Contacts.CONTENT_URI` | 监听通讯录变更 |
| Android（补充） | `CONTACT_LAST_UPDATED_TIMESTAMP` | 查询指定时间后变更 |

### 5.1.3 同步 API
| 接口 | 方法 | 路径 | 说明 |
|---|---|---|---|
| 双向同步 | POST | `/api/v1/sync` | 提交本地变更，获取服务端变更 |
| 获取变更增量 | GET | `/api/v1/sync/changes?since={ts}` | 拉取指定时间后变更 |
| 确认同步完成 | POST | `/api/v1/sync/ack` | 确认客户端已应用变更 |

### 5.2 验证标准（v0.2）
| 编号 | 测试场景 | 预期结果 | 优先级 |
|---|---|---|---|
| T-0201 | 设备 A 新增联系人→同步→设备 B 拉取 | 设备 B 可见新联系人，字段完整 | P0 |
| T-0202 | 设备 B 修改联系人→同步→设备 A 拉取 | 设备 A 可见修改内容 | P0 |
| T-0203 | 设备 A 删除联系人→同步→设备 B | 设备 B 对应联系人被删除 | P0 |

## 6. v0.3 冲突解决

### 6.1 冲突场景分类
| 冲突类型 | 场景描述 | 解决策略 |
|---|---|---|
| 字段级冲突 | 双方修改同一联系人不同字段 | 自动合并（各取最新字段值） |
| 值级冲突 | 双方修改同一字段为不同值 | 默认取最新时间戳，可手动选择 |
| 删除-修改冲突 | 一方删除，另一方修改同一联系人 | 提示用户选择保留或删除 |
| 新增重复 | 双方各自新增相似联系人 | 转入去重流程（v0.4） |

### 6.2 冲突解决流程
1. 服务端通过 `version + hash` 检测冲突
2. 写入 `conflict_log` 表
3. 返回冲突详情（双方版本对比）
4. 客户端展示冲突对比 UI，高亮差异字段
5. 用户选择：保留本地 / 保留云端 / 手动编辑合并
6. 提交解决结果并更新 `version`

### 6.3 历史版本
- 联系人每次变更保存历史快照
- 默认保留最近 30 天
- 付费用户可扩展至 365 天

### 6.4 验证标准（v0.3）
| 编号 | 测试场景 | 预期结果 | 优先级 |
|---|---|---|---|
| T-0301 | 双设备同时修改同一联系人不同字段 | 自动合并成功，各字段保留最新值 | P0 |
| T-0302 | 双设备同时修改同一字段为不同值 | 冲突提示 UI 正确展示，用户可选择 | P0 |
| T-0303 | 一方删除、另一方修改同一联系人 | 提示用户确认删除或保留 | P0 |
| T-0304 | 历史版本回滚 | 成功恢复至指定历史版本 | P1 |
| T-0305 | 批量冲突解决（20+ 条） | 支持一键全部保留本地/云端 | P1 |

## 7. v0.4 智能去重合并

### 7.1 重复检测算法
采用多级匹配，综合评分超过阈值判定重复。

| 匹配级别 | 匹配规则 | 权重 | 分值 |
|---|---|---|---|
| 精确匹配 | 电话完全一致（标准化后） | 40% | 100 |
| 精确匹配 | 邮箱完全一致（忽略大小写） | 30% | 100 |
| 模糊匹配 | 姓名相似度 > 85%（Jaro-Winkler） | 20% | 0-100 |
| 辅助匹配 | 公司名称相同 + 职位相似 | 10% | 0-100 |

- 阈值：综合评分 `>= 75` 判定“疑似重复”；`>= 90` 判定“高度重复”。

### 7.2 电话号码标准化
- 移除空格、破折号、括号
- 统一国际区号格式（E.164）
- 匹配时忽略区号前缀差异（如 `+86` 与 `0`）
- 推荐使用 `Google libphonenumber`

### 7.3 合并策略
| 字段 | 合并规则 |
|---|---|
| 姓名 | 保留更完整版本 |
| 电话号码 | 合并去重，保留全部非重复号码 |
| 邮箱地址 | 合并去重，保留全部非重复邮箱 |
| 公司/职位 | 保留最新更新值 |
| 地址 | 合并去重，保留全部非重复地址 |
| 备注 | 拼接合并并区分来源 |
| 头像 | 保留分辨率更高版本 |

### 7.4 用户交互流程
1. 系统后台扫描疑似重复联系人
2. 按置信度排序展示疑似重复组
3. 高度重复（>=90）：提示确认自动合并
4. 疑似重复（75-89）：展示详情供手动决策
5. 用户选择：合并 / 保留两者 / 忽略本次
6. 合并后生成统一联系人，原始记录存档

### 7.5 验证标准（v0.4）
| 编号 | 测试场景 | 预期结果 | 优先级 |
|---|---|---|---|
| T-0401 | 相同电话不同姓名 | 识别为重复，分数 >= 75 | P0 |
| T-0402 | 相同姓名不同电话 | 不判定为重复（除非其他字段匹配） | P0 |
| T-0403 | 国际区号格式差异（+86 vs 0） | 标准化后正确匹配 | P0 |
| T-0404 | 合并后数据完整性 | 所有非重复电话、邮箱均保留 | P0 |
| T-0405 | 批量去重（500+ 组） | 处理时间 < 30 秒，无误合并 | P1 |
| T-0406 | 用户拒绝合并后不再提示 | 忽略记录持久化，不重复提示 | P1 |

## 8. v1.0 正式发布

### 8.1 性能优化
| 指标 | 目标值 | 实现方案 |
|---|---|---|
| 首次全量同步（5000 联系人） | < 60 秒 | 分批并发上传 + gzip 压缩 |
| 增量同步响应时间 | < 3 秒 | Redis 缓存 + 增量查询索引 |
| 去重扫描（5000 联系人） | < 15 秒 | 预计算指纹 + 候选集索引 |
| API P99 延迟 | < 500ms | 查询优化 + 连接池 |
| 并发支持 | 1000 QPS | K8s 水平扩展 + 负载均衡 |

### 8.2 安全设计
| 安全领域 | 实现方案 |
|---|---|
| 传输加密 | TLS 1.3 + Certificate Pinning |
| 存储加密 | AES-256-GCM（电话、邮箱等敏感字段） |
| 认证机制 | JWT + Refresh Token，多设备管理 |
| 访问控制 | 基于用户 ID 的数据隔离与严格校验 |
| 数据脱敏 | 日志中电话/邮箱自动脱敏 |
| 审计日志 | 所有数据访问操作记录审计 |
| 备份策略 | 每日全量备份 + 实时 WAL 增量备份 |

### 8.3 监控与告警
- Prometheus：采集同步成功率、延迟、错误率
- Grafana：可视化看板监控系统健康
- PagerDuty：同步失败率 > 5% 触发告警
- 客户端：Crashlytics / Sentry 异常监控

### 8.4 备份与恢复
- 支持用户导出 vCard（`.vcf`）并导入其他应用
- 服务端支持用户级数据导出
- 账户注销后数据彻底删除，符合 GDPR 要求

## 11. 附录

### 11.1 同步状态机
- `local_only -> [上传] -> synced -> [本地修改] -> pending -> [同步] -> synced`
- `synced -> [双方修改] -> conflict -> [解决] -> synced`
- `synced -> [本地删除] -> deleted -> [同步] -> [删除]`

### 11.2 术语表
| 术语 | 说明 |
|---|---|
| 增量同步 | 仅同步上次同步后发生变化的联系人 |
| 全量同步 | 重新同步所有联系人，用于初始化或修复 |
| 乐观锁 | 基于版本号的并发控制机制 |
| 内容指纹 | 联系人内容的 SHA-256 哈希值 |
| E.164 | 国际电话号码标准格式（如 `+8613800138000`） |
| vCard | 通讯录交换标准（`.vcf`） |
| JWT | JSON Web Token，用于 API 身份认证 |
